# Foreign Future Protocol Support for uniffi-napi

## Problem

uniffi-napi cannot handle UniFFI's async foreign trait protocol. When Rust code calls an async method on a trait implemented in JavaScript (e.g., `AsyncFetcher.fetch()`), UniFFI passes a C function pointer (`ForeignFutureCallback`) and a struct-by-value argument (`ForeignFutureResult`) that uniffi-napi cannot currently process.

Three infrastructure gaps block this:

1. **C function pointers received as callback arguments** — UniFFI passes a `ForeignFutureComplete<T>` function pointer into the VTable callback. JS needs to receive this as a callable function and invoke it later.
2. **Struct-by-value passing** — The completion callback takes a `ForeignFutureResult<T>` struct by value (not by pointer). This requires `libffi::middle::Type::structure()` and marshalling JS objects into C struct byte layouts.
3. **Mutable struct out-pointers** — The `out_dropped_callback` parameter is `&mut ForeignFutureDroppedCallbackStruct`. This can be ignored for correctness (Rust null-checks before calling), with cancellation support wired up by the generated bindings layer, not by uniffi-napi.

## Design Constraints

- **Generic framework** — No library-specific Rust code. All new capabilities are declared via `register()`, matching the existing pattern.
- **Compatible with ubrn code generation** — The API must be callable from TypeScript generated by uniffi-bindgen-react-native (or a future napi backend). The generated code calls `uniffiFutureCallback.call(...)` with a plain JS object as the struct argument.
- **Mirrors UniFFI's FfiType system** — The `register()` API types map 1:1 from UniFFI's `FfiType` enum: `FfiType::Callback(name)` for function pointers, `FfiType::Struct(name)` for struct-by-value, `FfiType::Reference(Struct)` / `FfiType::MutReference(Struct)` for struct pointers.
- **No changes to uniffi-rs** — Works with UniFFI 0.31 as-is.

## Protocol Overview

The C-level protocol for an async VTable method (e.g., `AsyncFetcher.fetch`):

```
Rust calls VTable callback:
  fetch(handle: u64, input: RustBuffer,
        complete_cb: ForeignFutureCompleteRustBuffer,  // C function pointer
        cb_data: u64,                                   // opaque oneshot handle
        out_dropped_cb: &mut ForeignFutureDroppedCallbackStruct)
  → void

JS does async work, then calls:
  complete_cb(cb_data, ForeignFutureResult { return_value: RustBuffer, call_status: RustCallStatus })
```

The `complete_cb` is a monomorphized `foreign_future_complete::<T, UT>` — a dynamically-passed function pointer, not a named export. JS must receive it and call it later with a struct-by-value argument.

## Design

### Change 1: `FfiType.Callback` in callback arguments receives C function pointers

Currently, `FfiType.Callback('name')` in a VTable struct field creates a libffi closure (JS→C). The same `FfiType.Callback('name')` appearing as an **argument** to a VTable callback means the opposite: Rust passes a C function pointer into the JS callback.

The distinction is positional:
- **In a struct field definition**: creates a libffi closure wrapping a JS function (existing behavior)
- **In a callback argument list**: receives a C function pointer from Rust, wraps it as a callable JS function (new behavior)

When the trampoline encounters a `Callback`-typed argument in a VTable callback:
1. Read the raw function pointer from the C argument (`*mut c_void`)
2. Look up the callback's signature from the registered callback definitions
3. Create a JS function that, when called from JS, marshals its arguments and invokes the C function pointer via `libffi::middle::Cif::call()`

The wrapped JS function is passed to the VTable callback as a regular argument. The generated TypeScript calls it like any function: `uniffiFutureCallback.call(uniffiFutureCallback, uniffiCallbackData, resultObj)`.

### Change 2: `FfiType.Struct('name')` supports by-value structs

Currently, bare `FfiType.Struct('name')` panics in CIF construction — it's only used via `Reference(Struct(...))` for pointer passing. This change makes bare `Struct` work as a by-value type.

**Struct definition in `register()`:**

The `structs` section already exists for VTable struct definitions. The same section gains struct-by-value definitions. The struct field types determine whether it's a VTable (fields are `Callback` types) or a data struct (fields are scalar/RustBuffer/nested Struct types):

```js
structs: {
  // VTable struct (existing pattern — fields are callbacks)
  VTable_AsyncFetcher: [
    { name: 'uniffi_free', type: FfiType.Callback('callback_asyncfetcher_free') },
    { name: 'uniffi_clone', type: FfiType.Callback('callback_asyncfetcher_clone') },
    { name: 'fetch', type: FfiType.Callback('callback_asyncfetcher_fetch') },
  ],
  // Data struct (new — fields are value types)
  ForeignFutureResultRustBuffer: [
    { name: 'returnValue', type: FfiType.RustBuffer },
    { name: 'callStatus', type: FfiType.Struct('RustCallStatus') },
  ],
  RustCallStatus: [
    { name: 'code', type: FfiType.Int8 },
    { name: 'errorBuf', type: FfiType.RustBuffer },
  ],
  ForeignFutureDroppedCallbackStruct: [
    { name: 'callbackData', type: FfiType.UInt64 },
    { name: 'callback', type: FfiType.Callback('ForeignFutureDroppedCallback') },
  ],
}
```

**CIF construction** for `FfiTypeDesc::Struct(name)`:

Instead of panicking, look up the struct definition and build a `libffi::middle::Type::structure()` from its fields:
```rust
FfiTypeDesc::Struct(name) => {
    let fields = struct_defs.get(name);
    Type::structure(fields.iter().map(|f| ffi_type_for(&f.type_)).collect())
}
```

This requires passing the struct definitions into `ffi_type_for`, or pre-computing a map of struct name → libffi Type.

**Marshalling JS objects → C struct bytes:**

When a JS function (the wrapped `uniffiFutureCallback`) is called with a JS object argument at a `Struct`-typed position:
1. Look up the struct field definitions
2. For each field, extract the JS property by name and convert to C bytes
3. Write the bytes into a correctly-sized, correctly-aligned buffer matching the libffi struct layout
4. Pass the buffer to `libffi::middle::Cif::call()`

Nested structs (e.g., `callStatus: FfiType.Struct('RustCallStatus')` inside `ForeignFutureResultRustBuffer`) are handled recursively.

### Change 3: `FfiType.MutReference(Struct)` passes through as opaque pointer

For the `out_dropped_callback` parameter (`&mut ForeignFutureDroppedCallbackStruct`), the trampoline reads the mutable pointer and passes it to JS as an opaque value (or ignores it). Cancellation support is wired up by the generated bindings layer, not by uniffi-napi.

`MutReference` already maps to `Type::pointer()` in CIF construction. The trampoline just needs to not crash when it encounters this type as a callback argument — it can pass it through as a bigint (the raw pointer value) or as `undefined` if the JS side doesn't need it.

### Full register() example for AsyncFetcher

```js

  },
  functions: {
    [`uniffi_${CRATE}_fn_init_callback_vtable_asyncfetcher`]: {
      args: [FfiType.Reference(FfiType.Struct('VTable_AsyncFetcher'))],
      ret: FfiType.Void,
      hasRustCallStatus: false,
    },
    [`uniffi_${CRATE}_fn_func_use_async_fetcher`]: {
      args: [FfiType.Handle, FfiType.RustBuffer],
      ret: FfiType.Handle,
      hasRustCallStatus: true,
    },
    // ... poll/complete/free for rust_buffer (existing pattern)
  },
  callbacks: {
    callback_asyncfetcher_free: {
      args: [FfiType.UInt64],
      ret: FfiType.Void,
      hasRustCallStatus: false,
    },
    callback_asyncfetcher_clone: {
      args: [FfiType.UInt64],
      ret: FfiType.UInt64,
      hasRustCallStatus: false,
    },
    callback_asyncfetcher_fetch: {
      args: [
        FfiType.UInt64,                                              // uniffi_handle
        FfiType.RustBuffer,                                          // input

        FfiType.UInt64,                                              // callback_data
        FfiType.MutReference(                                        // out_dropped_cb
          FfiType.Struct('ForeignFutureDroppedCallbackStruct')
        ),
      ],
      ret: FfiType.Void,
      hasRustCallStatus: false,
    },
    ForeignFutureCompleteRustBuffer: {
      args: [FfiType.UInt64, FfiType.Struct('ForeignFutureResultRustBuffer')],
      ret: FfiType.Void,
      hasRustCallStatus: false,
    },
    ForeignFutureDroppedCallback: {
      args: [FfiType.UInt64],
      ret: FfiType.Void,
      hasRustCallStatus: false,
    },
  },
  structs: {
    VTable_AsyncFetcher: [
      { name: 'uniffi_free', type: FfiType.Callback('callback_asyncfetcher_free') },
      { name: 'uniffi_clone', type: FfiType.Callback('callback_asyncfetcher_clone') },
      { name: 'fetch', type: FfiType.Callback('callback_asyncfetcher_fetch') },
    ],
    ForeignFutureResultRustBuffer: [
      { name: 'returnValue', type: FfiType.RustBuffer },
      { name: 'callStatus', type: FfiType.Struct('RustCallStatus') },
    ],
    RustCallStatus: [
      { name: 'code', type: FfiType.Int8 },
      { name: 'errorBuf', type: FfiType.RustBuffer },
    ],
    ForeignFutureDroppedCallbackStruct: [
      { name: 'callbackData', type: FfiType.UInt64 },
      { name: 'callback', type: FfiType.Callback('ForeignFutureDroppedCallback') },
    ],
  },
});
```

### How the generated TypeScript uses this

The generated callback implementation (matching ubrn's `CallbackInterfaceImpl.ts` pattern):

```typescript
fetch: (uniffiHandle, input, uniffiFutureCallback, uniffiCallbackData, uniffiOutDroppedCallback) => {
    const uniffiMakeCall = async (signal: AbortSignal): Promise<string> => {
        const jsCallback = FfiConverterAsyncFetcher.lift(uniffiHandle);
        return await jsCallback.fetch(FfiConverterString.lift(input), { signal });
    };

    const uniffiHandleSuccess = (returnValue: string) => {
        uniffiFutureCallback.call(
            uniffiFutureCallback,
            uniffiCallbackData,
            {
                returnValue: FfiConverterString.lower(returnValue),
                callStatus: { code: 0 }
            }
        );
    };

    const uniffiHandleError = (code: number, errorBuf: Uint8Array) => {
        uniffiFutureCallback.call(
            uniffiFutureCallback,
            uniffiCallbackData,
            {
                returnValue: new Uint8Array(0),
                callStatus: { code, errorBuf }
            }
        );
    };

    return uniffiTraitInterfaceCallAsync(
        uniffiMakeCall,
        uniffiHandleSuccess,
        uniffiHandleError,
        FfiConverterString.lower
    );
},
```

The key point: `uniffiFutureCallback` arrives as a callable JS function (wrapping the C pointer). The struct argument `{ returnValue, callStatus }` is a plain JS object that uniffi-napi marshals into C struct bytes when the function is called.

## Implementation Scope

### In scope
- `FfiType.Callback` as callback argument → wraps C function pointer as callable JS function
- `FfiType.Struct` by-value → `libffi::middle::Type::structure()` + JS object → C struct marshalling
- Nested struct support (e.g., `RustCallStatus` inside `ForeignFutureResult`)
- `FfiType.MutReference(Struct)` as callback argument → pass-through (opaque pointer or ignored)

### Out of scope
- Cancellation support (`ForeignFutureDroppedCallbackStruct` write-back) — wired by generated bindings, not uniffi-napi
- `RustBuffer` fields inside by-value structs that need ownership transfer — the completion callback consumes the struct, so Rust takes ownership of the RustBuffer data
- Async foreign traits with error returns — same mechanism, different `ForeignFutureResult` variant
